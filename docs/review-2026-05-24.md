## Security

### 1. [High] `.envrc` missing from `.gitignore`

- File: `.gitignore`
- Lines: (whole file)
- A direnv user creating `.envrc` with `export GEMINI_API_KEY=sk-...` followed by `git add -A` would commit the key into this public MIT repo. The pre-existing GitHub Python template ignored `.envrc`; the rewrite dropped it.
- **Fix:** add `.envrc` to `.gitignore`. Optional belt-and-suspenders: add a README line warning that `.env` and `.envrc` must never be committed.

### 2. [Medium] HTML escape into `<script type="application/json">` blocks

- File: `src/hubed_audio_learn/templates/html/player.html.j2`
- Lines: ~48–49 (the `timing-data` and `markers-data` `<script>` tags)
- `timing_json` and `markers_json` are rendered with `| safe`. `pydantic.model_dump_json` does not HTML-escape `<`, so any LLM-supplied string in `markers` (figure_ids) containing the substring `</script>` terminates the script block, breaks `JSON.parse`, and runs any injected HTML/JS that follows.
- **Fix:** before passing to the template, do `json_str.replace("</", "<\\/")` (the standard JSON-in-HTML escape). The browser still parses the JSON correctly; the closing tag can no longer escape.

## Bugs / correctness

### 3. [High] Claude JSON responses wrapped in markdown fences not unwrapped

- Files: `src/hubed_audio_learn/providers/llm/claude.py:43`, and consumers in `pipeline/{research,outline,script}.py`
- `result.stdout.strip()` returns the raw output. Claude `--print` commonly wraps JSON in ```json ... ``` even when prompted not to; `json.loads(raw)` fails both retry attempts and the stage raises `RuntimeError: ... failed after 2 attempts`. This is the most likely first-run blocker.
- **Fix:** in `ClaudeProvider.generate`, after `.strip()`, strip a leading ```json (or ```) fence and a trailing ``` if present. Alternatively pull a `_extract_json` helper used by all three stages that finds the outermost balanced `{...}` substring.

### 4. [Medium] Gemini response shape accessed unconditionally

- File: `src/hubed_audio_learn/providers/tts/gemini.py:124`
- `response.candidates[0].content.parts[0].inline_data.data` has no guards. Safety filtering, quota errors, or a text-only response raises `IndexError`/`AttributeError` with no diagnostic context. The TTS stage has no retry wrapper, so a single bad call kills the run.
- **Fix:** check `response.candidates`, `parts`, and `inline_data is not None`; raise a `RuntimeError` that includes `response.candidates[0].finish_reason` (and the prompt feedback if present) so the user sees why TTS refused.

### 5. [Medium] `run_package` silently overwrites prior episodes

- File: `src/hubed_audio_learn/pipeline/package.py` (whole `run_package`)
- Lines: ~88+
- Filenames are deterministic (`<date>-<slug>.{ext}`). Same-day same-topic re-run clobbers a prior good episode; Syncthing/Obsidian-Sync propagates the delete to the phone. There's no warning, backup, or `--force` requirement.
- **Fix:** if `final_mp3` exists, either suffix with `-2`, `-3`, … or refuse and require `--force` (and document the choice in `audio-learn doctor` output if helpful).

### 6. [Medium] Silence-detection heuristic for per-turn timing is fragile

- File: `src/hubed_audio_learn/providers/tts/gemini.py:47–50`
- `_recover_turn_timings` trusts `len(intervals) == num_turns` as a one-to-one mapping. In practice merged adjacent same-speaker turns plus internal sentence pauses can coincidentally produce a matching count with misaligned segments. The char-proportional fallback would be safer in those cases.
- **Fix options:** (a) require both `len(intervals) == num_turns` and that detected intervals roughly match expected per-turn character budgets (within e.g. 50%) before trusting silence detection; otherwise fall back. (b) Long-term: send turns to TTS one at a time and concatenate, which eliminates the heuristic entirely. (c) Re-evaluate when Gemini exposes per-turn offsets natively.

### 7. [Medium] No referential integrity between script and outline

- File: `src/hubed_audio_learn/pipeline/html_render.py:55–56` (and the schemas/script stage in general)
- The LLM-emitted script can reference a `slide_marker.figure_id` that doesn't exist in the outline. `_ordered_figures` silently skips it; `_markers_map` and `package.figures_referenced` still record it. The HTML player then tries to activate a missing slide id, so the previous slide stays on screen during the wrong segment, but the manifest claims the figure was referenced.
- **Fix:** in `pipeline/script.py`, after `Script.model_validate`, cross-check every `turn.slide_marker.figure_id` exists in the outline's figures (and `section_ref` exists in sections). Treat a mismatch like a JSON validation failure: feed the error back into the retry prompt.

### 8. [Low] `audio_duration_estimate_seconds` mislabeled

- File: `src/hubed_audio_learn/pipeline/package.py:61`
- The field stores `timing.turns[-1].start_seconds` — the start of the final turn, not the audio duration. A 10:30 episode whose final turn begins at 9:55 shows 595s in the manifest while the MP3 is ~630s.
- **Fix:** either rename the field to `last_turn_start_seconds` (cheap), or derive real duration from the MP3 (load with pydub and read `len(segment) / 1000`).

### 9. [Low] `subprocess.run` runs without explicit `encoding="utf-8"`

- File: `src/hubed_audio_learn/providers/llm/claude.py:27–32`
- `subprocess.run(..., text=True, input=prompt)` inherits the locale's default encoding. Windows (cp1252) or non-UTF-8 Linux locales raise `UnicodeEncodeError` on prompts containing em-dashes, accented names from PDF text, etc. The global CLAUDE.md explicitly calls out cross-platform compatibility.
- **Fix:** pass `encoding="utf-8"` to `subprocess.run`. Optionally also set `errors="replace"` on stdout decoding so non-UTF-8 model output doesn't crash.

### 10. [Low] `package.py` HTML rewrite is fragile to template changes

- File: `src/hubed_audio_learn/pipeline/package.py:98`
- `html.replace('src="episode.mp3"', f'src="{base}.mp3"')` only matches the exact double-quoted literal. Any future template tweak (single quotes, attribute reorder, extra whitespace) silently no-ops the rewrite; the packaged HTML ends up referencing a sibling `episode.mp3` that doesn't exist.
- **Fix options:** (a) parametrize the template with the final filename and call `html_render` after `package` knows the slug. (b) use a small regex `re.sub(r'src="[^"]*episode\.mp3"', f'src="{base}.mp3"', html, count=1)`. (c) Add a packaging test that asserts the rewrite happened.

## Out of scope for v1 but noted

- Timestamp `_format_timestamp` lacks an hour roll-over (`[125:50]` for a 2h-5m episode). Phase-1 `long` targets ~30–40 min so this rarely fires; revisit if `long` is ever pushed past 60 min.
- `arbitrary_types_allowed=True` on `Config` (`src/hubed_audio_learn/config.py:12`) is unnecessary now that `Path` is natively supported by pydantic v2 — cosmetic only.
- `voice_id` carries a `gemini:` prefix in Phase 1 even though only one TTS provider exists. Leave for Phase 2 when Dia (`dia:S1`) is added.
